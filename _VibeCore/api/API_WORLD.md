# API World — World Definition

> 本文件定義 **API 作為一個「世界概念」的本質角色**。
>
> 本文件不描述實作、不描述協定、不描述傳輸，
> 只回答一個問題：
>
> 👉 **API 在世界中「是什麼」，而不是「怎麼做」。**

---

## 1. API 的世界本質（World Essence）

在本世界體系中，**API 並不是系統能力，也不是行為執行者**。

API 的世界本質定義為：

> **API 是「世界狀態與世界意圖的對外語意表達界面」。**

換言之：

* API 不擁有世界狀態
* API 不推動世界流程
* API 不裁決世界結果

API 僅負責：

* 將「已存在於世界內部的語意」
* 以「可被外部理解的形式」呈現或輸入

---

## 2. API 與世界主體的關係（Relation to the World）

世界的主體已由世界核心定義，並由既定機制維持：

* 世界狀態由 Container 所承載
* 世界流程由 Lifecycle 所裁決
* 世界路徑由 Router 所限制

在此結構下：

### API 的位置

API **永遠位於世界主體之外**，其角色是：

* 世界的「對外窗口（Boundary Interface）」
* 世界語意的「轉譯點（Semantic Projection）」

API 不在世界內部發生行為，而是：

* 接收世界允許的輸入意圖
* 或暴露世界已成立的狀態結果

---

## 3. API 不是能力（Not a Capability）

根據既有世界規則：

* 能力（Capability）是可被 Container 管理、注入、組合的世界構件

API **不符合能力定義**，原因如下：

* API 不具備可被注入的行為
* API 不應影響模組可用性
* API 不應成為能力授權的判斷依據

因此：

> **API 不得被註冊為能力，不得出現在能力清單中。**

---

## 4. API 不具備世界裁決權（No World Authority）

世界裁決權已在核心中被明確規範。

API World 明確聲明：

* API 不裁決「是否允許」
* API 不裁決「是否有效」
* API 不裁決「是否成功」

所有裁決結果：

* 均必須已由世界內部完成
* API 僅能轉述結果，不得加工裁決語意

---

## 5. API 的存在條件（Existence Condition）

API 只有在以下條件成立時，才被視為「合法存在於世界中」：

* 世界已進入允許對外互動的階段
* 該互動不會觸發新的世界裁決
* 該互動不會繞過既定世界機制

若上述任一條件不成立：

> API 不應存在，或其呼叫應被視為世界外部噪音。

---

## 6. 世界視角下的 API 分類前導（Preview）

在世界語意層，API 將被區分為兩種根本類型：

* **Query**：

  * 對外揭露「已成立的世界狀態」

* **Command**：

  * 對外提交「意圖」，但不保證世界是否接受

此分類僅為世界語意前導，
其完整定義將於 **API_CATEGORIES.md** 中說明。

---

## 7. 世界不允許的 API 語意（Explicit Prohibitions）

以下語意在世界中被明確禁止：

* API 直接建立、銷毀或切換世界
* API 決定模組是否啟用
* API 作為世界流程的起點或主控器
* API 隱含任何「只要呼叫就會成功」的語意

任何出現上述語意的 API 描述：

> **視為違反世界規則，而非實作錯誤。**
