# 世界補完建議（範本對照）

本文件是「建議」而非世界規則。它根據 `_workspace/範本/src` 的目錄與語意角色，對照既有世界規則，指出目前世界文件尚未覆蓋或仍模糊的概念邊界，僅作為世界層補完的候選方向，是否納入須由世界裁決。

---

## 觀察到的結構語意（僅描述，不裁決）

範本結構包含明確的「平台啟動與生命週期」區塊、容器與能力註冊區塊、API 互動邊界、路由與守衛、模組集合、以及樣式與佈局的分層。這些結構顯示出平台層的協調責任、模組層的職責隔離、以及世界運行中對狀態、權限與視覺一致性的需求。

---

## 與既有世界規則的對照缺口（需要被明文化的語意）

以下項目已在範本中以結構形式出現，但在世界層仍未形成可裁決的語意描述，容易造成工程落地時的解讀歧義：

1) 平台啟動與生命週期的操作責任邊界  
世界已定義生命週期階段，但「平台層如何協調啟動、恢復、登入與模組組裝」的角色界限尚未被明確描述，導致工程層可能以實作細節填補世界語意。

2) 容器與能力註冊在平台層的最小責任  
世界已聲明 Container 是唯一能力入口，但平台層如何建立核心能力、如何保證一次性註冊、以及如何避免模組越權，仍缺少可裁決的描述。

3) 路由、守衛與世界可見性的語意邊界  
世界已聲明 Routing 為最後防線，但「平台層如何建構路由」「模組如何對外輸出路由」「守衛如何承接世界可見性裁決」在世界層仍未形成可依循的語意敘述。

4) 平台配置與世界上下文的關係  
世界已宣告 PlatformConfig 的優先性，但「平台配置的來源」「載入失敗時的世界語意」「平台配置對模組可用性的影響」尚未被世界層明文化。

5) UI 佈局與樣式責任的層級分工  
範本顯示存在全域樣式、Layout 層與模組內樣式，但世界層未描述這些層的語意角色與邊界，容易讓工程層自行裁決其優先關係與責任範圍。

## 補完方向（只列出候選位置與意圖，不新增規則）

若世界設計者決定補完，可考慮將語意補充至下列既有世界文件中，維持世界治理的一致性：

- `01_WORLD_MODEL.md`：補充「平台協調層」與「模組輸出入口」在世界中的語意角色。  
- `04_WORLD_LIFECYCLE.md`：補充「平台啟動、恢復、組裝」的語意邊界與可觀測性。  
- `06_STRUCTURE_AND_NAMING.md`：補充「layout / styles / router / app-lifecycle」在世界層的結構語意責任（僅描述角色，不涉實作）。  
- `03_WORLD_API_RULES.md`：補充 API client 與平台層行為的語意界線（避免工程層誤作裁決）。  

以上均為候選方向，非變更提案。是否進入 Change Protocol 必須由世界裁決。

## 風險提醒（世界一致性角度）

若上述語意長期未被世界層明文化，工程層必然會以實作補位，導致同一個結構在不同 Project 中被不同解讀。這會削弱世界的可裁決性與一致性，並使世界層難以約束後續擴張。
