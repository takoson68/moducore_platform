# API Categories — World Semantic Classification

> 本文件定義 **API 在世界語意層的根本分類方式**。
>
> 此分類**不是工程設計模式**，也不是實作建議，
> 而是用來回答：
>
> 👉 世界如何理解「一個 API 呼叫在語意上想做什麼」。

---

## 1. 分類的世界前提（World Premise）

世界核心已明確指出：

* 世界狀態的成立，必須經由世界內部機制完成
* 世界裁決不對外開放
* 世界流程不可被外部直接驅動

因此，API 在世界中的角色只能落在兩個語意方向之一：

* **揭露已成立的狀態**
* **提交尚未裁決的意圖**

這不是技術選擇，而是世界必然結果。

---

## 2. Query API（世界狀態查詢）

### 2.1 世界定義

**Query** 的世界語意定義為：

> 對外揭露「世界中已成立、已被裁決的狀態」。

Query 並不改變世界，只是讓世界被看見。

---

### 2.2 世界語意特性

Query 具有以下不可變語意：

* 不引發世界狀態改變
* 不觸發任何世界裁決
* 不推動世界流程
* 呼叫成功 ≠ 取得想要的資料（仍受世界狀態限制）

Query 的結果：

* 只能來自「已存在」的世界狀態
* 不得包含推論、補齊或隱性變形

---

### 2.3 世界禁止事項

以下行為 **禁止** 出現在 Query 語意中：

* 因查詢而建立狀態
* 因查詢而修正狀態
* 因查詢而觸發流程
* 以 Query 之名行 Command 之實

任何上述情況：

> 視為世界語意違規，而非效能或工程問題。

---

## 3. Command API（世界意圖提交）

### 3.1 世界定義

**Command** 的世界語意定義為：

> 對外提交一個「希望世界發生改變的意圖」，
> 但不保證世界會接受、執行或成功。

Command 是請求，不是指令。

---

### 3.2 世界語意特性

Command 具有以下世界語意特性：

* Command 本身不改變世界
* 世界是否改變，取決於內部裁決結果
* Command 的完成，不等於狀態已成立
* Command 必須可被世界拒絕

換言之：

> Command 只是一個「世界可能會忽略的聲音」。

---

### 3.3 Command 與結果的斷裂

在世界語意中：

* Command ≠ 成功
* Command ≠ 狀態成立
* Command ≠ 流程完成

任何將 Command 與結果直接綁定的描述：

> 都屬於世界語意污染。

---

## 4. Query 與 Command 的嚴格分離

### 4.1 不可混用原則

世界規則要求：

* 一個 API **只能屬於一種語意分類**
* 不存在「又查又改」的合法語意

若一個 API 同時：

* 揭露狀態
* 又試圖引發改變

則該 API 在世界中 **不存在合法定位**。

---

### 4.2 拒絕模糊分類

以下分類方式在世界語意層**不被接受**：

* 「部分修改」
* 「輕量更新」
* 「帶副作用查詢」
* 「同步狀態指令」

這些用語掩蓋了世界裁決的存在。

---

## 5. 世界對分類的治理價值（Why It Matters）

此分類存在的目的不是限制工程，而是：

* 防止 API 被誤用為世界快捷鍵
* 防止工程層繞過世界裁決
* 讓世界行為保持可推理性
* 讓 AI Agent 不會誤以為 API 是世界主體
